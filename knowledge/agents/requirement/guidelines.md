# 需求分析知识库

## 需求分析方法

### 5W1H 分析法

| 维度 | 问题 | 示例 |
|------|------|------|
| **Who** | 谁是用户？ | 设备管理员、运维人员 |
| **What** | 需要什么功能？ | 设备信息管理、状态跟踪 |
| **When** | 什么时候使用？ | 日常工作、设备巡检时 |
| **Where** | 在哪里使用？ | 办公室、移动端、现场 |
| **Why** | 为什么需要？ | 提高管理效率、减少错误 |
| **How** | 如何实现？ | CRUD 操作、批量导入 |

### MoSCoW 优先级

| 优先级 | 含义 | 示例 |
|--------|------|------|
| **M**ust | 必须有 | 设备 CRUD、用户登录 |
| **S**hould | 应该有 | 批量导入、数据导出 |
| **C**ould | 可以有 | 统计图表、数据可视化 |
| **W**on't | 暂不做 | 移动端 APP、多语言 |

### INVEST 原则

好的用户故事应该满足：

| 原则 | 含义 | 检查点 |
|------|------|--------|
| **I**ndependent | 独立的 | 不依赖其他故事 |
| **N**egotiable | 可协商的 | 细节可讨论 |
| **V**aluable | 有价值的 | 对用户有价值 |
| **E**stimable | 可估算的 | 能评估工作量 |
| **S**mall | 足够小 | 一个迭代能完成 |
| **T**estable | 可测试的 | 有明确的验收标准 |

## 需求分类

### 功能性需求

```
1. 业务功能
   - 核心业务流程
   - 数据处理规则
   - 业务计算逻辑

2. 系统功能
   - 用户认证授权
   - 数据导入导出
   - 日志审计

3. 接口需求
   - API 接口
   - 第三方集成
   - 数据同步
```

### 非功能性需求

| 类型 | 指标 | 示例 |
|------|------|------|
| **性能** | 响应时间、吞吐量 | 页面加载 < 2s，支持 100 并发 |
| **安全** | 认证、授权、加密 | JWT 认证，HTTPS 传输 |
| **可用性** | 正常运行时间 | 99.9% 可用性 |
| **可扩展性** | 扩展能力 | 支持横向扩展 |
| **兼容性** | 浏览器、设备 | Chrome/Firefox/Safari |

## 需求文档结构

### 用户需求文档 (URD)

```markdown
# 用户需求文档

## 1. 项目背景
- 项目来源
- 业务背景
- 目标用户

## 2. 用户需求列表
| 编号 | 需求描述 | 优先级 | 来源 |
|------|----------|--------|------|
| UR-001 | 作为管理员，我想管理设备信息 | Must | 会议记录 |

## 3. 业务流程图
（业务流程描述）

## 4. 验收标准
（验收条件）
```

### 软件需求规格 (SRS)

```markdown
# 软件需求规格说明书

## 1. 引言
- 目的、范围、定义

## 2. 总体描述
- 产品视角、功能、约束

### 3.1 功能需求
### 3.2 非功能需求
### 3.3 接口需求

## 4. 附录
- 术语表、参考文档
```

## 需求优先级评估

### 价值-复杂度矩阵

```
高价值 · 低复杂度 → 优先做 (Quick Wins)
高价值 · 高复杂度 → 规划做 (Major Projects)
低价值 · 低复杂度 → 可以做 (Fill-ins)
低价值 · 高复杂度 → 不要做 (Time Sinks)
```

### Kano 模型

| 类型 | 描述 | 示例 |
|------|------|------|
| **基本型** | 必须满足 | 登录、数据保存 |
| **期望型** | 越多越好 | 搜索速度、界面美观 |
| **兴奋型** | 超出预期 | 智能推荐、一键操作 |

## 常见需求陷阱

### ❌ 避免的问题

1. **需求镀金**
   - 过度设计功能
   - 解决不存在的问题

2. **需求蔓延**
   - 无控制的需求增加
   - 边界不清晰

3. **模糊需求**
   - "系统要快"
   - "界面要好看"
   - 缺少可量化标准

4. **遗漏需求**
   - 忽略非功能需求
   - 忽略边界条件

### ✅ 正确的做法

1. **需求追溯**
   - 记录需求来源
   - 建立追溯矩阵

2. **需求评审**
   - 多方参与评审
   - 确认理解一致

3. **需求变更控制**
   - 变更评审流程
   - 影响范围评估
