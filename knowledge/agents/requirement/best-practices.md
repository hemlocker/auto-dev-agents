# 需求分析最佳实践

## 需求收集技巧

### 多渠道收集

```
1. 用户访谈
   - 一对一深度访谈
   - 关注真实场景和痛点

2. 问卷调查
   - 大规模数据收集
   - 统计分析需求优先级

3. 观察
   - 现场观察用户操作
   - 发现隐性需求

4. 原型验证
   - 快速原型演示
   - 收集用户反馈

5. 数据分析
   - 用户行为数据
   - 发现需求趋势
```

### 需求访谈技巧

```markdown
## 好的提问方式

✅ "您在工作中最常遇到什么问题？"
✅ "这个功能对您的工作有多重要？"
✅ "如果没有这个功能，您会怎么做？"
✅ "您期望这个功能多长时间响应？"

## 避免的提问方式

❌ "您需要这个功能吗？"（引导性问题）
❌ "这个功能好不好？"（主观判断）
❌ "您觉得系统应该怎么做？"（技术决策转嫁）
```

## 需求分析流程

### 标准流程

```
1. 需求收集
   └→ 多渠道收集原始需求

2. 需求分类
   └→ 功能需求 / 非功能需求 / 约束

3. 需求分析
   └→ 可行性、优先级、依赖关系

4. 需求规格化
   └→ 文档化、标准化

5. 需求验证
   └→ 评审、确认

6. 需求管理
   └→ 变更控制、追溯
```

### 需求评审检查清单

```markdown
## 完整性检查
- [ ] 所有功能需求都有描述
- [ ] 非功能需求有量化指标
- [ ] 业务流程完整
- [ ] 异常情况已考虑

## 一致性检查
- [ ] 需求之间无冲突
- [ ] 术语使用一致
- [ ] 优先级合理

## 可行性检查
- [ ] 技术可行
- [ ] 资源可行
- [ ] 时间可行

## 可测试性检查
- [ ] 需求可验证
- [ ] 验收标准明确
```

## 需求优先级决策

### 价值-成本分析

```
           高价值
              │
    优先做    │   规划做
   (Quick     │  (Major
    Wins)     │ Projects)
──────────────┼──────────────
   可以做     │   不要做
  (Fill-ins)  │ (Time Sinks)
              │
           低价值
    低成本 ─────────── 高成本
```

### RICE 评分法

| 因素 | 说明 | 计算方式 |
|------|------|----------|
| **R**each | 覆盖用户数 | 用户数量 |
| **I**mpact | 影响程度 | 1-5 分 |
| **C**onfidence | 确定性 | 50%-100% |
| **E**ffort | 工作量 | 人天 |

**RICE 得分** = (Reach × Impact × Confidence) / Effort

## 需求变更管理

### 变更流程

```
变更请求 → 影响分析 → 评审决策 → 实施变更 → 验证确认
    │           │           │
    │           │           └→ 接受/拒绝/延后
    │           └→ 范围、成本、时间影响
    └→ 记录变更原因和内容
```

### 变更评估要点

```markdown
## 影响范围评估

1. 功能影响
   - 影响哪些功能模块
   - 是否影响现有功能

2. 进度影响
   - 增加多少工作量
   - 是否影响里程碑

3. 成本影响
   - 人力成本
   - 资源成本

4. 质量影响
   - 是否引入风险
   - 测试范围变化
```

## 非功能需求定义

### 性能需求量化

| 指标 | 定义 | 示例 |
|------|------|------|
| 响应时间 | 用户操作到系统响应的时间 | < 2秒 |
| 吞吐量 | 单位时间处理的事务数 | 100 TPS |
| 并发数 | 同时在线用户数 | 1000 用户 |
| 资源占用 | CPU、内存使用率 | CPU < 70% |

### 安全需求定义

```markdown
## 认证与授权
- 用户认证方式: JWT / OAuth / SAML
- 密码强度要求: 8位+数字+特殊字符
- 会话超时: 30分钟无操作自动登出

## 数据安全
- 传输加密: HTTPS/TLS 1.2+
- 存储加密: AES-256
- 敏感数据脱敏: 日志、报表中脱敏显示

## 审计日志
- 登录日志: 记录登录时间、IP
- 操作日志: 记录关键操作
- 数据变更日志: 记录数据修改
```

## 常见问题与解决方案

### 问题1: 需求模糊

**表现**: "系统要快"、"界面要好用"

**解决**:
```markdown
❌ 模糊: "系统要快"
✅ 明确: "页面加载时间 < 2秒，API 响应 < 500ms"

❌ 模糊: "界面要好用"
✅ 明确: "用户完成核心流程 < 5 步，新手培训时间 < 30 分钟"
```

### 问题2: 需求镀金

**表现**: 过度设计，功能超出实际需要

**解决**:
- 严格按 MoSCoW 优先级
- 每个 "Could" 需求都要有充分理由
- 问："不做这个功能会怎样？"

### 问题3: 需求蔓延

**表现**: 需求不断增加，范围失控

**解决**:
- 建立变更控制流程
- 每个变更都要评估影响
- 设置需求冻结点

## 需求文档质量标准

### SMART 原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **S**pecific | 具体明确 | 而非"系统要快" |
| **M**easurable | 可度量 | "响应时间 < 2秒" |
| **A**chievable | 可实现 | 技术上可行 |
| **R**elevant | 相关性 | 与业务目标相关 |
| **T**ime-bound | 有时限 | "在 v1.0 版本实现" |

## 总结

| 场景 | 最佳实践 |
|------|----------|
| 需求收集 | 多渠道收集，避免单一来源 |
| 需求分析 | 5W1H + MoSCoW + INVEST |
| 优先级决策 | 价值-成本矩阵 + RICE 评分 |
| 变更管理 | 建立流程，评估影响 |
| 文档质量 | SMART 原则，量化指标 |
