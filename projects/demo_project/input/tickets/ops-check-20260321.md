# 运维检查问题报告

## 发现的问题

### 问题 1: 缺少健康检查端点
- **严重程度**: 中
- **类别**: 监控
- **描述**: 后端服务未实现 `/api/health` 健康检查端点，无法进行有效的服务健康监控
- **影响**: 监控系统无法准确判断服务状态，可能导致故障发现延迟
- **建议**: 在 backend 中添加 `/api/health` 端点，返回数据库连接状态和服务运行状态
- **状态**: 待处理

### 问题 2: 日志配置不完善
- **严重程度**: 中
- **类别**: 日志
- **描述**: 
  - docker-compose.yml 未配置日志轮转策略
  - 未指定日志驱动和大小限制
  - 长期运行可能导致磁盘空间耗尽
- **影响**: 日志文件无限增长，可能占用全部磁盘空间
- **建议**: 在 docker-compose.yml 中为每个服务添加 logging 配置：
  ```yaml
  logging:
    driver: "json-file"
    options:
      max-size: "10m"
      max-file: "5"
  ```
- **状态**: 待处理

### 问题 3: 数据库备份策略缺失
- **严重程度**: 高
- **类别**: 数据安全
- **描述**: 未配置数据库自动备份机制，数据持久化仅依赖 Docker volume
- **影响**: 如果 Docker volume 损坏或误删除，将导致数据永久丢失
- **建议**: 
  1. 添加定时备份脚本，每日备份数据库文件
  2. 将备份文件存储到独立位置或云存储
  3. 定期测试备份恢复流程
- **状态**: 待处理

### 问题 4: 缺少资源限制配置
- **严重程度**: 中
- **类别**: 资源管理
- **描述**: docker-compose.yml 未配置容器资源限制（CPU、内存）
- **影响**: 单个容器可能占用过多资源，影响其他服务运行
- **建议**: 为每个服务添加 deploy.resources 配置：
  ```yaml
  deploy:
    resources:
      limits:
        cpus: '1.0'
        memory: 512M
      reservations:
        cpus: '0.5'
        memory: 256M
  ```
- **状态**: 待处理

### 问题 5: 安全配置不足
- **严重程度**: 中
- **类别**: 安全
- **描述**: 
  - 后端服务直接暴露 3000 端口（虽然有 expose 限制）
  - 未配置 HTTPS
  - 未设置 CORS 策略
- **影响**: 存在潜在的安全风险，生产环境不应使用 HTTP
- **建议**: 
  1. 生产环境必须配置 HTTPS 证书
  2. 在 nginx.conf 中配置 CORS 策略
  3. 移除不必要的端口暴露
- **状态**: 待处理

### 问题 6: 缺少重启策略
- **严重程度**: 低
- **类别**: 可用性
- **描述**: docker-compose.yml 未配置容器重启策略
- **影响**: 容器异常退出后不会自动重启，需要手动干预
- **建议**: 为每个服务添加 restart 策略：
  ```yaml
  restart: unless-stopped
  ```
- **状态**: 待处理

### 问题 7: 监控和告警缺失
- **严重程度**: 中
- **类别**: 监控
- **描述**: 未配置任何监控和告警系统
- **影响**: 无法及时发现和响应系统故障
- **建议**: 部署 Prometheus + Grafana + Alertmanager 监控栈（配置详见 output/operations/监控配置.md）
- **状态**: 待处理

### 问题 8: 环境变量管理不当
- **严重程度**: 低
- **类别**: 配置管理
- **描述**: 敏感信息（如数据库密码、API 密钥）直接写在 docker-compose.yml 中
- **影响**: 配置文件泄露会导致敏感信息暴露
- **建议**: 
  1. 使用 .env 文件管理环境变量
  2. 将 .env 文件加入 .gitignore
  3. 使用 Docker secrets 管理敏感信息
- **状态**: 待处理
